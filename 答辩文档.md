## 一、项目简介

### 1.1 项目名称

**CampusBite（校园美食地图）**

### 1.2 项目背景

大学生每天都要吃饭，但校园周边的店铺那么多，哪家好吃？哪家性价比高？目前没有一个专门面向本校学生的美食评价平台。大众点评信息太杂，不够聚焦。所以我想做一个**只收录校园周边店铺**的美食评价系统，让同学们可以：

- 浏览校园周边的美食店铺
- 按分类快速找到想吃的类型
- 看其他同学的评价和评分
- 自己也发表评价，分享用餐体验

### 1.3 功能清单

| 功能 | 页面 | 说明 |
|------|------|------|
| 首页展示 | `/` | Hero 横幅 + 分类导航 + 高分推荐 + 最新收录 |
| 分类浏览 | `/shops/` | 左侧分类筛选 + 右侧店铺列表 |
| 店铺详情 | `/shop/<id>/` | 店铺图片、信息、评分、评价列表 |
| 发表评价 | `/shop/<id>/review/` | 登录后选择评分 + 填写评价 |
| 关键词搜索 | `/search/?q=关键词` | 按名称、描述、位置模糊搜索 |
| 数据统计 | `/analytics/` | 评分分布柱状图 + 分类占比饼图 |
| 判别分析 | `/taste/` | 分析用户口味偏好 + 个性化推荐 |
| 后台管理 | `/admin/` | Django Admin 管理所有数据 |

## 二、技术选型

### 2.1 用了什么技术

| 技术 | 版本 | 作用 |
|------|------|------|
| **Python** | 3.12 | 编程语言 |
| **Django** | 6.0 | Web 框架（后端） |
| **SQLite** | 内置 | 数据库 |
| **Bootstrap** | 5.3 | UI 框架（前端样式） |
| **Chart.js** | 4.4 | 图表库（数据可视化） |

### 2.2 为什么选这些技术（面试可能问）

**问：为什么用 Django 而不用 Flask？**

> Django 自带 Admin 后台、ORM 数据库操作、模板引擎，开箱即用。Flask 更轻量但很多东西要自己搭。对于课设项目，Django 能更快完成，而且 Admin 后台省了很多工作量。

**问：为什么用 SQLite 而不用 MySQL？**

> SQLite 是 Python 自带的，不需要额外安装和配置数据库服务。课设数据量小，SQLite 完全够用。如果以后要部署上线，只需要改一行配置就能切换到 MySQL。

**问：为什么用 Bootstrap 而不用 Vue/React？**

> 这个项目是服务端渲染（SSR），页面由 Django 在服务器生成 HTML 返回给浏览器。Bootstrap 是 CSS 框架，直接在 HTML 里写 class 就能出效果，不需要前端构建工具。Vue/React 是前端框架，适合做复杂交互的单页应用（SPA），这个项目用不上。

**问：为什么用 Chart.js 而不用 ECharts？**

> Chart.js 更轻量（~200KB），通过 CDN 引入一行代码就能用。ECharts 功能更强大但体积更大。课设场景下 Chart.js 够用了。

## 三、数据模型设计

### 3.1 模型关系图

```
┌──────────────┐        ┌──────────────┐        ┌──────────────┐
│   Category   │  1   N │     Shop     │  1   N │    Review    │
│   （分类）    │────────│   （店铺）    │────────│   （评价）    │
│──────────────│        │──────────────│        │──────────────│
│ id           │        │ id           │        │ id           │
│ name（分类名）│        │ name（店名）  │        │ shop_id（外键）│
│ icon（图标）  │        │ category_id  │        │ user_id（外键）│
│ created_at   │        │ image（图片） │        │ rating（评分） │
└──────────────┘        │ location     │        │ content      │
                        │ description  │        │ created_at   │
                        │ avg_price    │        └──────────────┘
                        │ created_at   │
                        └──────────────┘
```

### 3.2 三个模型详解

#### Category（分类）

```python
class Category(models.Model):
    name = models.CharField(max_length=50)       # 分类名称，如"食堂窗口"
    icon = models.CharField(max_length=50)        # Bootstrap 图标类名，如"bi-building"
    created_at = models.DateTimeField(auto_now_add=True)  # 创建时间（自动填充）
```

**作用**：给店铺分组，比如"食堂窗口"、"奶茶饮品"、"小吃快餐"等。

**`icon` 字段**：存的是 Bootstrap Icons 的类名，前端直接用 `<i class="bi bi-building">` 就能显示图标，不需要上传图片。

#### Shop（店铺）

```python
class Shop(models.Model):
    name = models.CharField(max_length=100)       # 店铺名称
    category = models.ForeignKey(Category)         # 外键：属于哪个分类
    image = models.ImageField(upload_to='shops/')  # 店铺图片
    location = models.CharField(max_length=200)    # 位置，如"第一食堂二楼"
    description = models.TextField(blank=True)     # 描述（可选填）
    avg_price = models.DecimalField(max_digits=8, decimal_places=1)  # 人均价格
    created_at = models.DateTimeField(auto_now_add=True)

    def avg_rating(self):
        """计算平均评分（Python 层面计算）"""
        reviews = self.reviews.all()
        if reviews:
            return round(sum(r.rating for r in reviews) / reviews.count(), 1)
        return 0

    def review_count(self):
        """评价总数"""
        return self.reviews.count()
```

**外键 (ForeignKey)**：`category` 字段关联到 Category 表，表示这家店属于哪个分类。`on_delete=CASCADE` 意思是：如果分类被删除，该分类下的店铺也一起删除。

**`ImageField`**：Django 的图片上传字段，用户上传的图片保存到 `media/shops/` 目录。需要安装 Pillow 库。

**`avg_rating()` 方法**：这是一个**计算属性**，每次调用时实时计算。不是数据库字段，所以不会存储在数据库里。

#### Review（评价）

```python
class Review(models.Model):
    shop = models.ForeignKey(Shop, related_name='reviews')  # 外键：评价的是哪家店
    user = models.ForeignKey(User)                           # 外键：谁写的评价
    rating = models.IntegerField(choices=[(i, f'{i}星') for i in range(1, 6)])
    content = models.TextField()                             # 评价内容
    created_at = models.DateTimeField(auto_now_add=True)
```

**`related_name='reviews'`**：这个很关键！它让我们可以通过 `shop.reviews.all()` 反向查询到某家店的所有评价。如果不设置 related_name，默认是 `shop.review_set.all()`。

**`choices` 参数**：限制 rating 只能填 1-5，在 Admin 后台会显示为下拉选择框。

**外键到 User**：`User` 是 Django 自带的用户模型（`django.contrib.auth.models.User`），我们复用了 Django 的认证系统，不需要自己建用户表。

## 四、功能详解

### 4.1 首页（`/`）

**功能**：展示分类导航、高分推荐、最新收录。

**视图代码**：

```python
def home(request):
    categories = Category.objects.all()                              # 所有分类
    top_shops = Shop.objects.annotate(                               # 注解：计算平均分
        avg_r=Avg('reviews__rating')                                 # 跨表聚合
    ).order_by('-avg_r')[:8]                                         # 降序取前8
    new_shops = Shop.objects.order_by('-created_at')[:4]             # 最新4家
    return render(request, 'food/home.html', {...})
```

**关键技术**：
- `annotate(avg_r=Avg('reviews__rating'))`：这是 Django ORM 的**聚合查询**，在数据库层面计算每家店的平均评分，而不是在 Python 里循环计算。`reviews__rating` 是跨表查询语法（双下划线），意思是"通过 Review 表的 rating 字段"。
- `order_by('-avg_r')`：按平均分降序排列（负号表示降序）。
- `[:8]`：切片取前 8 条，对应 SQL 的 `LIMIT 8`。

### 4.2 店铺列表页（`/shops/`）

**功能**：展示所有店铺，支持按分类筛选。

```python
def shop_list(request):
    category_id = request.GET.get('category')     # 从 URL 参数获取分类 ID
    shops = Shop.objects.all()                     # 默认查询所有店铺
    if category_id:                                 # 如果有筛选参数
        shops = shops.filter(category_id=category_id)  # 过滤
    return render(request, 'food/shop_list.html', {...})
```

**URL 参数**：访问 `/shops/?category=1` 时，`request.GET.get('category')` 就能拿到 `1`。

**`filter()` 方法**：相当于 SQL 的 `WHERE category_id = 1`。

### 4.3 店铺详情页（`/shop/<id>/`）

**功能**：展示店铺详情 + 评价列表。

```python
def shop_detail(request, pk):
    shop = get_object_or_404(Shop, pk=pk)    # 获取店铺，不存在返回 404
    reviews = shop.reviews.all()              # 通过 related_name 反向查询所有评价
    return render(request, 'food/shop_detail.html', {'shop': shop, 'reviews': reviews})
```

**`get_object_or_404()`**：如果找不到对象，自动返回 404 页面，不需要手写 `if` 判断。

**`shop.reviews.all()`**：因为 Review 模型设置了 `related_name='reviews'`，所以可以通过店铺对象直接查到所有评价。

### 4.4 搜索功能（`/search/`）

**功能**：按关键词搜索店铺名称、描述、位置。

```python
def search(request):
    q = request.GET.get('q', '')
    shops = Shop.objects.filter(
        Q(name__icontains=q) |               # 名称包含关键词
        Q(description__icontains=q) |        # 描述包含关键词
        Q(location__icontains=q)             # 位置包含关键词
    ) if q else Shop.objects.none()
```

**`Q` 对象**：Django 的查询对象，可以用 `|`（或）、`&`（与）、`~`（非）组合多个条件。这里用 `|` 表示三个字段任意一个匹配就行。

**`icontains`**：不区分大小写的模糊匹配，相当于 SQL 的 `LIKE '%关键词%'`。

### 4.5 发表评价（`/shop/<id>/review/`）

**功能**：登录用户可以发表评价。

```python
@login_required                                    # 装饰器：必须登录才能访问
def add_review(request, pk):
    shop = get_object_or_404(Shop, pk=pk)
    if request.method == 'POST':                   # 只处理 POST 请求
        rating = request.POST.get('rating')        # 获取表单数据
        content = request.POST.get('content')
        if rating and content:                     # 验证非空
            Review.objects.create(                 # 创建评价
                shop=shop, user=request.user,
                rating=int(rating), content=content
            )
            messages.success(request, '评价发表成功！')
    return redirect('shop_detail', pk=pk)          # 重定向回详情页
```

**`@login_required`**：Django 内置的装饰器，未登录用户访问会自动跳转到登录页面。

**POST-PRG 模式**：表单提交后用 `redirect()` 重定向，而不是直接渲染页面。这样用户刷新页面不会重复提交表单。

**`messages.success()`**：Django 的消息框架，设置一条成功提示，下一个页面会自动显示。

### 4.6 数据统计页（`/analytics/`）

**功能**：用图表展示评分分布和分类占比。

```python
def analytics(request):
    # 评分分布：统计 1-5 星各有多少条评价
    rating_dist = [0] * 5
    for row in Review.objects.values('rating').annotate(count=Count('id')):
        rating_dist[row['rating'] - 1] = row['count']

    # 分类占比：统计各分类有多少家店铺
    pie_rows = Shop.objects.values('category__name').annotate(count=Count('id'))
```

**`values('rating').annotate(count=Count('id'))`**：相当于 SQL 的 `SELECT rating, COUNT(id) FROM review GROUP BY rating`，按评分分组计数。

**前端渲染**：数据通过 `json.dumps()` 转成 JSON 字符串传给模板，再用 Chart.js 渲染图表。

### 4.7 判别分析页（`/taste/`）

**功能**：分析用户的评分记录，判断口味偏好，推荐店铺。

**口味判别逻辑**：

```python
# 1. 统计用户在各分类的评分
for review in reviews:
    cat_name = review.shop.category.name
    category_ratings[cat_name].append(review.rating)

# 2. 计算各分类平均分
for cat, ratings in category_ratings.items():
    cat_avg[cat] = round(sum(ratings) / len(ratings), 1)

# 3. 生成口味标签
if user_avg_rating >= 4.5:
    taste_tags.append('宽容型食客')        # 平均分高 → 宽容
elif user_avg_rating >= 3.5:
    taste_tags.append('理性型食客')        # 中等 → 理性
else:
    taste_tags.append('挑剔型食客')        # 平均分低 → 挑剔

# 4. 推荐：找用户偏好分类中未评价的高分店铺
preferred_cats = [cat for cat, avg in sorted_cats if avg >= 3.5]
rec_shops = Shop.objects.filter(
    category__name__in=preferred_cats
).exclude(
    id__in=reviewed_shop_ids               # 排除已评价的
).annotate(avg_r=Avg('reviews__rating')).order_by('-avg_r')[:6]
```

**推荐算法**：
1. 找出用户评分 >= 3.5 的分类（偏好分类）
2. 在这些分类中，排除用户已经评价过的店铺
3. 按其他用户的平均评分降序排列
4. 取前 6 家推荐

## 五、核心代码讲解

### 5.1 模板继承

Django 的模板系统支持**继承**，避免重复代码。

**基模板** `base.html`：
```html
<!DOCTYPE html>
<html>
<body>
    <nav>导航栏（所有页面共用）</nav>
    <main>
        {% block content %}{% endblock %}   ← 子模板填充这里
    </main>
    <footer>页脚（所有页面共用）</footer>
</body>
</html>
```

**子模板** `home.html`：
```html
{% extends 'includes/base.html' %}          ← 继承基模板

{% block content %}                          ← 填充 content 区域
    <h1>首页内容</h1>
{% endblock %}
```

**好处**：改导航栏只需要改 `base.html` 一个文件，所有页面自动更新。

### 5.2 ORM 查询（数据库操作）

Django 的 ORM 让我们用 Python 代码操作数据库，不需要写 SQL。

| Python 代码 | 等价 SQL |
|-------------|---------|
| `Shop.objects.all()` | `SELECT * FROM shop` |
| `Shop.objects.filter(category_id=1)` | `SELECT * FROM shop WHERE category_id = 1` |
| `Shop.objects.get(pk=1)` | `SELECT * FROM shop WHERE id = 1` |
| `Shop.objects.create(name='xxx')` | `INSERT INTO shop (name) VALUES ('xxx')` |
| `shop.save()` | `UPDATE shop SET ... WHERE id = 1` |
| `shop.delete()` | `DELETE FROM shop WHERE id = 1` |
| `Shop.objects.annotate(avg_r=Avg('reviews__rating'))` | `SELECT *, AVG(r.rating) AS avg_r FROM shop LEFT JOIN review ... GROUP BY shop.id` |

### 5.3 外键和反向查询

```python
# 正向查询：从 Review 找 Shop
review = Review.objects.first()
print(review.shop.name)          # 这条评价对应的店铺名称

# 反向查询：从 Shop 找 Review（通过 related_name）
shop = Shop.objects.first()
reviews = shop.reviews.all()     # 这家店的所有评价
count = shop.reviews.count()     # 评价总数
```

### 5.4 Q 对象（复杂查询）

```python
# 普通 filter 只能 AND
Shop.objects.filter(name='xxx', category_id=1)   # WHERE name='xxx' AND category_id=1

# Q 对象可以 OR
Shop.objects.filter(
    Q(name__icontains='麻辣') | Q(location__icontains='食堂')
)   # WHERE name LIKE '%麻辣%' OR location LIKE '%食堂%'
```

### 5.5 用户认证

Django 自带完整的用户认证系统，我们直接复用：

- `request.user`：当前登录的用户对象
- `@login_required`：登录校验装饰器
- `User.objects.create_superuser()`：创建管理员
- 模板中 `{% if user.is_authenticated %}` 判断是否登录

## 八、答辩常见问题及回答

### 基础问题

**Q：这个项目用了什么技术栈？**

> 后端用 Django 6.0 + SQLite，前端用 Bootstrap 5 + Django Template，图表用 Chart.js。Django 负责处理业务逻辑和数据库操作，Bootstrap 负责页面样式，Chart.js 负责数据可视化。

**Q：数据库有几张表？分别是什么？**

> 3 张表。Category（分类表）、Shop（店铺表）、Review（评价表）。Category 和 Shop 是一对多关系（一个分类下有多个店铺），Shop 和 Review 也是一对多关系（一家店有多个评价）。另外还复用了 Django 自带的 User 用户表。

**Q：外键是什么？为什么要用外键？**

> 外键是表与表之间的关联关系。比如 Shop 表的 `category` 字段是外键，指向 Category 表，表示这家店属于哪个分类。用外键可以保证数据一致性，比如不能给一个不存在的分类添加店铺。同时支持跨表查询，比如可以查"食堂窗口"分类下的所有店铺。

### 进阶问题

**Q：搜索功能是怎么实现的？**

> 用 Django 的 Q 对象实现。用户输入关键词后，系统同时匹配店铺名称、描述、位置三个字段，任意一个匹配就返回。用 `icontains` 实现不区分大小写的模糊匹配。Q 对象用 `|`（或）连接，相当于 SQL 的 OR 条件。

**Q：评分是怎么计算的？有没有性能问题？**

> 有两种计算方式。一种是模型里的 `avg_rating()` 方法，在 Python 层面循环计算，适合详情页单家店的场景。另一种是视图里的 `annotate(avg_r=Avg('reviews__rating'))`，在数据库层面用 SQL 聚合函数计算，适合列表页需要排序的场景。后者性能更好，因为数据库计算比 Python 循环快，而且避免了 N+1 查询问题。

**Q：什么是 N+1 查询问题？**

> 假设有 100 家店铺，如果用循环计算评分：
> ```python
> for shop in Shop.objects.all():           # 1 次查询
>     avg = shop.reviews.aggregate(Avg('rating'))  # 每家店 1 次，共 100 次
> ```
> 总共 101 次查询。用 `annotate` 只需要 1 次 SQL 就搞定了。

**Q：用户认证是怎么做的？**

> 直接复用了 Django 自带的 User 模型和认证系统。`@login_required` 装饰器检查用户是否登录，未登录自动跳转登录页。`request.user` 可以获取当前登录用户。模板中用 `{% if user.is_authenticated %}` 判断登录状态。没有自己写注册登录页面，用的是 Django Admin 的登录页。

**Q：推荐算法是什么？**

> 分三步：第一步，统计用户在各分类的评分，计算每个分类的平均分。第二步，找出平均分 >= 3.5 的分类作为用户偏好。第三步，在偏好分类中排除用户已评价的店铺，按其他用户的平均评分降序取前 6 家。简单说就是"推荐你喜欢的分类中你没去过的好店"。

**Q：判别分析的口味标签是怎么生成的？**

> 基于三个维度：一是偏好分类，看用户在哪个分类评分最高；二是总体宽容度，根据用户所有评价的平均分判断（>=4.5 宽容，3.5-4.5 理性，<3.5 挑剔）；三是评分稳定性，用标准差衡量（<=0.8 稳定，>1.5 波动大）。三个维度的标签组合起来就是用户的口味画像。

### 部署相关

**Q：怎么部署上线？**

> 课设阶段用的是 Django 自带的开发服务器（`python manage.py runserver`）。如果要上线，需要用 Gunicorn 或 uWSGI 作为应用服务器，Nginx 作为反向代理，数据库换成 MySQL 或 PostgreSQL。静态文件需要用 `collectstatic` 收集，图片上传需要配置媒体文件服务。

**Q：SQLite 和 MySQL 有什么区别？什么时候用 MySQL？**

> SQLite 是文件数据库，数据存在一个文件里，不需要安装数据库服务，适合开发和小型项目。MySQL 是服务端数据库，支持并发访问、事务、主从复制，适合生产环境。Django 的 ORM 屏蔽了数据库差异，切换只需要改 `settings.py` 里的一行配置。
